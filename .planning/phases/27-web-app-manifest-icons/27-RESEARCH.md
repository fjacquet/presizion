## Summary

Phase 27 installs the web app manifest and all required icons to make Presizion installable to iOS and Android home screens. The work is entirely static file and HTML meta-tag work — no runtime JavaScript changes, no React component changes, no new runtime npm dependencies.

The critical constraint is GitHub Pages subpath deployment at `/presizion/`. Every URL in the manifest (`start_url`, `scope`, icon `src` paths) must include this prefix. Omitting it causes the installed icon to open a 404. This is the single highest-risk pitfall (M-3) and must be verified with Chrome DevTools Application tab before the phase is complete.

The second critical constraint is iOS Safari: it ignores the manifest's `icons` array entirely. A separate `<link rel="apple-touch-icon" href="/presizion/icons/apple-touch-icon-180x180.png">` in `index.html` is mandatory (M-4). Without it, iOS shows a screenshot of the page as the home screen icon.

**Primary recommendation:** Write `public/manifest.webmanifest` manually (no vite-plugin-pwa), generate icons once using `@vite-pwa/assets-generator` CLI from `public/favicon.svg`, and add five meta/link tags to `index.html`. Total scope is small: one JSON file, one config file, one script run, and five HTML lines.

### Pattern 1: Web App Manifest with GitHub Pages Subpath

**What:** A static JSON file at `public/manifest.webmanifest` with all paths explicitly including `/presizion/`.

**When to use:** Always — the manifest must be complete before any installability testing.

**Example:**
```json
{
  "name": "Presizion — Cluster Refresh Sizing",
  "short_name": "Presizion",
  "description": "Size a refreshed server cluster from existing environment metrics",
  "start_url": "/presizion/",
  "scope": "/presizion/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#3B82F6",
  "icons": [
    {
      "src": "/presizion/icons/pwa-192x192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/presizion/icons/pwa-512x512.png",
      "sizes": "512x512",
      "type": "image/png"
    },
    {
      "src": "/presizion/icons/maskable-512x512.png",
      "sizes": "512x512",
      "type": "image/png",
      "purpose": "maskable"
    }
  ]
}
```

Key points:
- `start_url` and `scope` use absolute `/presizion/` prefix (not relative `./`)
- Icon `src` values use absolute paths with `/presizion/` prefix
- `theme_color` is `#3B82F6` — matches favicon.svg's `fill="#3B82F6"` blue
- `background_color` is `#ffffff` — matches light mode splash screen
- File extension `.webmanifest` (not `.json`) — preferred MIME type is `application/manifest+json`

### Pattern 2: index.html Meta Tags for iOS + Android

**What:** Five additions to `<head>` in `index.html`.

**When to use:** All five tags must be added together — each serves a distinct browser/OS.

**Example:**
```html
<!-- manifest link — Chrome/Android reads this -->
<link rel="manifest" href="/presizion/manifest.webmanifest" />

<!-- iOS Safari: ignores manifest, reads these directly -->
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="default" />
<meta name="apple-mobile-web-app-title" content="Presizion" />
<link rel="apple-touch-icon" sizes="180x180" href="/presizion/icons/apple-touch-icon-180x180.png" />

<!-- Android Chrome address bar color + iOS PWA splash (dual for light/dark) -->
<meta name="theme-color" content="#ffffff" media="(prefers-color-scheme: light)" />
<meta name="theme-color" content="#0a0a0a" media="(prefers-color-scheme: dark)" />

<!-- Viewport: add viewport-fit=cover for iOS safe-area CSS variables -->
<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover" />
```

Current `index.html` viewport tag is `width=device-width, initial-scale=1.0` — replace it with the version that includes `viewport-fit=cover`.

### Pattern 3: Icon Generation Config

**What:** `pwa-assets.config.ts` at project root using `minimal2023Preset`.

**When to use:** Run once; regenerate only if favicon.svg changes.

```typescript
// Source: https://github.com/vite-pwa/assets-generator
import { defineConfig, minimal2023Preset } from '@vite-pwa/assets-generator/config'

export default defineConfig({
  preset: minimal2023Preset,
  images: ['public/favicon.svg'],
})
```

Add to `package.json` scripts:
```json
"generate-icons": "pwa-assets-generator --config pwa-assets.config.ts"
```

Run: `npm run generate-icons` — produces all four PNGs in `public/icons/`.

The `minimal2023Preset` generates exactly:
- `pwa-192x192.png` — standard Android/Chrome icon
- `pwa-512x512.png` — large standard icon
- `maskable-512x512.png` — Android adaptive launcher with safe-zone padding
- `apple-touch-icon-180x180.png` — iOS home screen icon (solid background, no transparency)

### Anti-Patterns to Avoid

- **Relative URLs in manifest:** `"src": "icons/icon-192.png"` resolves against the manifest URL (`/presizion/manifest.webmanifest`), which happens to work here, but `start_url: "/"` will break silently at launch.
- **Using vite-plugin-pwa:** Installs service worker infrastructure; conflicts with Vite 8 build output.
- **Transparent icon PNGs:** iOS fills transparent areas with black — use solid background. The `minimal2023Preset` handles this automatically.
- **`apple-mobile-web-app-status-bar-style: black-translucent`:** Places content under the iPhone notch/Dynamic Island — requires explicit `env(safe-area-inset-top)` CSS handling. Use `"default"` instead.
- **Single `theme-color` without media query:** Use two meta tags with `media="(prefers-color-scheme: light|dark)"` to match the app's 3-way theme toggle.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-size icon generation | ImageMagick scripts, custom Node canvas scripts | `@vite-pwa/assets-generator` | Handles maskable safe-zone padding, correct PNG compression, all required sizes in one command |

**Key insight:** Icon generation is a one-time build step, not runtime code. The complexity is in knowing which sizes/formats are required by which platform, not in the generation itself.

## Common Pitfalls

### Pitfall 1: Manifest start_url/scope Missing the Base Path (M-3 — CRITICAL)

**What goes wrong:** `"start_url": "/"` causes the installed home screen icon to open `https://fjacquet.github.io/` (a 404) instead of `https://fjacquet.github.io/presizion/`.

**Why it happens:** Copy-paste from tutorials that assume root deployment. GitHub Pages project sites always live under a repository path.

**How to avoid:** Set both `"start_url": "/presizion/"` and `"scope": "/presizion/"` explicitly. Verify in Chrome DevTools → Application → Manifest → Installability — "Start URL is in scope" must show green.

**Warning signs:** Chrome DevTools shows "Start URL is not in scope" error. Installing to home screen and tapping the icon shows a 404.

### Pitfall 2: iOS Safari Ignores Manifest Icons (M-4 — CRITICAL)

**What goes wrong:** Icons defined in `manifest.webmanifest` are ignored by iOS Safari. "Add to Home Screen" shows a page screenshot.

**Why it happens:** Apple's longstanding non-standard behavior — Safari requires `<link rel="apple-touch-icon">` in HTML.

**How to avoid:** Add `<link rel="apple-touch-icon" sizes="180x180" href="/presizion/icons/apple-touch-icon-180x180.png" />` to `index.html`. Icon must be a 180x180 PNG with solid (non-transparent) background.

**Warning signs:** Lighthouse PWA audit passes (it only checks the manifest) but iOS home screen shows a screenshot.

### Pitfall 3: apple-mobile-web-app-status-bar-style Without apple-mobile-web-app-capable (M-5)

**What goes wrong:** The status-bar-style meta tag has no effect unless paired with `apple-mobile-web-app-capable: yes`.

**How to avoid:** Always add both tags together. Use `"default"` for status-bar-style — not `"black-translucent"` (which obscures content under the notch without additional CSS).

### Pitfall 4: Theme Color Mismatch

**What goes wrong:** `theme_color` in the manifest uses a purple/violet value from earlier research (`#7c3aed`) but the actual app brand color (as established by `favicon.svg`) is blue `#3B82F6`.

**How to avoid:** The favicon.svg has `fill="#3B82F6"`. Use `#3B82F6` as `theme_color` in the manifest and as the light-mode `theme-color` meta. Dark mode meta should be `#0a0a0a` matching the dark background.

**Warning signs:** The browser address bar or PWA splash screen shows a different color than the app icon.

### Pitfall 5: Icons Not in Build Output

**What goes wrong:** Generated icons sit in `public/icons/` during development but fail to appear at `dist/presizion/icons/` in production.

**Why it happens:** Vite copies the `public/` directory verbatim to `dist/` with the base path prefix. If icons are generated into a location outside `public/`, they are not included.

**How to avoid:** Always generate into `public/icons/`. After `npm run build`, verify `dist/presizion/icons/pwa-192x192.png` exists before deploying.

### Pitfall 6: Stale Manifest After Reinstall

**What goes wrong:** Browser caches the old manifest. Even after updating `manifest.webmanifest`, the installed PWA still uses old values (wrong icon, wrong theme color).

**How to avoid:** This is a testing concern, not a code concern. Uninstall the PWA from the home screen and reinstall after each manifest change during testing. Chrome DevTools Application → Manifest → "Update" button forces a refresh.

## Code Examples

Verified patterns from official sources:

### Complete index.html head section (after changes)
```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />
    <!-- Updated: added viewport-fit=cover for iOS safe area -->
    <meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover" />
    <title>Presizion — Cluster Refresh Sizing</title>

    <!-- Web App Manifest -->
    <link rel="manifest" href="/presizion/manifest.webmanifest" />

    <!-- iOS Safari home screen -->
    <meta name="apple-mobile-web-app-capable" content="yes" />
    <meta name="apple-mobile-web-app-status-bar-style" content="default" />
    <meta name="apple-mobile-web-app-title" content="Presizion" />
    <link rel="apple-touch-icon" sizes="180x180" href="/presizion/icons/apple-touch-icon-180x180.png" />

    <!-- Theme color (Chrome address bar, Android splash) — dual for light/dark -->
    <meta name="theme-color" content="#ffffff" media="(prefers-color-scheme: light)" />
    <meta name="theme-color" content="#0a0a0a" media="(prefers-color-scheme: dark)" />

    <!-- Existing theme init script — keep as-is -->
    <script>
      try {
        var s = localStorage.getItem('presizion-theme')
        if (s === 'dark' || (!s && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
          document.documentElement.classList.add('dark')
        }
      } catch (_) {}
    </script>
  </head>
```

Note: The favicon `href="/favicon.svg"` (without `/presizion/` prefix) is intentional — it references the Vite dev server root, not the deployed path. Vite handles this during build. The manifest and apple-touch-icon links must use the full `/presizion/` prefix because they are embedded as-is in the deployed HTML.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `manifest.json` | `manifest.webmanifest` | Spec matured ~2020 | Correct MIME type; `application/manifest+json`; both work but `.webmanifest` is preferred |
| Single `theme-color` meta | Two `theme-color` meta tags with `media` attribute | Chrome 93+ (2021) | Adapts address bar color to user's system color scheme preference |
| Full PWA plugin (e.g., vite-plugin-pwa) for icons | Standalone `@vite-pwa/assets-generator` CLI | 2023 | Decouples icon generation from service worker infrastructure |
| Requiring service worker for installability | Service worker NOT required for Chrome installability | Chrome 89+ (2021) | Manifest with `display: standalone` + icons is sufficient; no SW needed |

**Deprecated/outdated:**
- `"display": "fullscreen"`: Hides status bar on iOS — avoid; use `"standalone"` which keeps status bar visible
- `apple-touch-icon-precomposed`: Older Apple format; iOS ignores the `precomposed` distinction since iOS 7 — use `apple-touch-icon` only
- `browserconfig.xml` (Windows tiles): Internet Explorer/Edge legacy; not needed for modern browsers

## Open Questions

1. **favicon.svg icon crop for 192x192**
   - What we know: `favicon.svg` has a 32x32 viewBox with a blue square background and white P mark. The `minimal2023Preset` will upscale this to 192x192 and 512x512.
   - What's unclear: Whether the upscaled result looks crisp or pixelated at large sizes (SVG-to-PNG rasterization should be clean, but worth visual inspection).
   - Recommendation: Run `npm run generate-icons`, visually inspect the output, and proceed — no code change needed either way.

2. **theme-color for dark mode splash screen**
   - What we know: The manifest has a single `theme_color` field (no light/dark variant). The `background_color` is used for the splash screen.
   - What's unclear: Whether to use `#3B82F6` (brand blue) or `#0a0a0a` (dark bg) as the single manifest `theme_color`.
   - Recommendation: Use `#3B82F6` in the manifest (brand color for splash screen background); use the dual `<meta name="theme-color">` tags in HTML to adapt the address bar per system preference.
